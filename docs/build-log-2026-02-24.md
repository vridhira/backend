**Service methods:**

### 2. Meilisearch Integration + Unified Admin Search UI (`cfee189`)

Added Meilisearch as a second search provider and replaced the Algolia-only admin page
with a unified **Search Engine** management page covering both providers plus a
`"default"` (Medusa built-in) fallback mode.

#### 2a. Meilisearch Module — `src/modules/meilisearch/`

| File | Purpose |
|------|---------|
| `service.ts` | `MeilisearchModuleService` — wraps the `meilisearch` npm client |
| `index.ts` | Module definition + `MEILISEARCH_MODULE` export key |

**Service methods:**

| Method | Description |
|--------|-------------|
| `indexData(data, type)` | Upserts product documents; sets `id` as primary key |
| `retrieveFromIndex(ids, type)` | Fetches documents by ID (silently skips missing) |
| `deleteFromIndex(ids, type)` | Removes documents by ID |
| `applySettings(features)` | Calls `index.updateSettings()` — applies searchable / filterable / sortable attributes + typo tolerance live |
| `searchProducts(query)` | Executes a search, returns hits |

**Graceful startup:** if `MEILISEARCH_HOST`, `MEILISEARCH_API_KEY`, or
`MEILISEARCH_PRODUCT_INDEX_NAME` are not set, the module logs a warning and loads
in an inactive state. Methods throw a `MedusaError` if called without credentials.
This means both Algolia and Meilisearch modules can be registered in `medusa-config.ts`
simultaneously — only the active provider is ever called.

**env vars required:**
```
MEILISEARCH_HOST=http://localhost:7700
MEILISEARCH_API_KEY=your-meilisearch-master-key
MEILISEARCH_PRODUCT_INDEX_NAME=products
```

Self-hosted: <https://www.meilisearch.com/docs/learn/getting_started/installation>  
Cloud: <https://cloud.meilisearch.com>

#### 2b. Search Config — `src/lib/search-config.ts`

Central config store that persists runtime settings to `.search-config.json` at
the project root (created automatically on first use).

**Config shape:**
```json
{
  "activeProvider": "default",
  "algoliaFeatures": { "typoTolerance": true, "highlights": true, "analytics": false, "searchableAttributes": ["title", "description", "handle"] },
  "meilisearchFeatures": { "typoTolerance": true, "faceting": true, "highlighting": true, "searchableAttributes": ["title", "description", "handle"], "filterableAttributes": ["categories.name", "tags.value", "status"], "sortableAttributes": ["title"] }
}
```

`activeProvider` values: `"algolia"` | `"meilisearch"` | `"default"`

**Exports:**
- `readSearchConfig()` — reads `.search-config.json`, returns defaults if file absent
- `writeSearchConfig(partial)` — deep-merges `partial` and writes back
- `getActiveProvider()` — convenience wrapper returning `config.activeProvider`

#### 2c. Updated Subscribers — Provider Routing

`product-sync.ts` and `product-delete.ts` now call `getActiveProvider()` on every event
and route to the correct workflow:

| Provider | Create/Update routes to | Delete routes to |
|----------|------------------------|------------------|
| `"algolia"` | `syncProductsWorkflow` | `deleteProductsFromAlgoliaWorkflow` |
| `"meilisearch"` | `syncProductsMeilisearchWorkflow` | `deleteProductsFromMeilisearchWorkflow` |
| `"default"` | _(no-op)_ | _(no-op)_ |

`meilisearch-sync.ts` subscriber: listens on `meilisearch.sync` custom event,
paginates all products in batches of 50 and calls `syncProductsMeilisearchWorkflow`.

### 3. Bug Fixes (`438ac2a`, `9663526`)

#### Fix: `meilisearch` ESM-only package in CJS backend

**Problem:** `meilisearch` npm v0.38+ ships as an ESM-only package. The static
`import { Meilisearch } from "meilisearch"` compiled to `require("meilisearch")` in the
CJS Medusa backend, causing a runtime error.

**Solution:** Replaced the static top-level import with a lazy `async createClient()`
method that uses dynamic `import()`:
```typescript
private async createClient(): Promise<MeiliClient> {
  const { Meilisearch } = await import("meilisearch")
  return new Meilisearch({ host, apiKey }) as unknown as MeiliClient
}
```
Local `MeiliClient` / `MeiliIndex` interface types defined in the file remove the
need for any compile-time import from `meilisearch`. The client promise is pre-warmed
in the constructor so the first method call does not pay the import cost.

#### Fix: TypeScript excessive stack depth in `sync-products.ts`

**Problem:** The `transform()` callback in `src/workflows/sync-products.ts` had an
implicit parameter type, causing TypeScript to recursively expand
`WorkflowData<Product[]>` union types until it hit the "excessive stack depth" limit.

**Solution:** Added an explicit type annotation to the callback parameter:
```typescript
(data: { products: { id: string; status: string; [key: string]: unknown }[] }) => {
```
