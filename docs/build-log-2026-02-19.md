## ✅ Features Implemented

### 1. Cash on Delivery (COD) Payment Provider

Built a production-grade custom MedusaJS v2 payment provider for COD with fraud controls
appropriate for the Indian market.

**Location:** `vridhira-marketplace/src/modules/cod-payment/`

**Key behaviour:**
- Validates order amount against configurable limits (default: ₹100 min, ₹50,000 max)
- Caps new customer orders at ₹1,500 to reduce first-order fraud risk
- Limits each customer to 3 COD orders per day
- Flags orders above ₹3,000 for OTP verification (threshold configurable)
- No external API calls — session managed locally
- Registered in `medusa-config.ts` under the `payment` module as provider ID `cod`

**Configurable limits (all in paise):**

| Option | Default | Meaning |
|--------|---------|---------|
| `min_order_amount` | 10000 | ₹100 minimum |
| `max_order_amount` | 5000000 | ₹50,000 maximum |
| `max_daily_orders` | 3 | orders per customer per day |
| `new_customer_limit` | 150000 | ₹1,500 cap for first-time buyers |
| `otp_threshold` | 300000 | ₹3,000 — orders above this require OTP |

### 4. Auto-Shipment on Order Placement

Built a `order.placed` subscriber that automatically sends every new order to Shiprocket
without requiring admin action.

**Location:** `vridhira-marketplace/src/subscribers/order-placed.ts`

**Flow:**
```
Customer places order
        ↓
order.placed event fires
        ↓
Subscriber checks Shiprocket is configured
        ↓
Detects COD vs Prepaid from payment provider
        ↓
Builds Shiprocket order payload (billing, shipping, items, weight/dimensions)
        ↓
Creates Shiprocket order → assigns cheapest courier → generates AWB → schedules pickup
```

This is fire-and-forget: fulfillment errors are logged server-side without failing
or rolling back the order — the customer's order confirmation is never affected.

### 8. Razorpay `refund.processed` Webhook → Refund Email

Migrated the Razorpay webhook from the legacy Express-style handler to a proper
MedusaJS v2 route at `src/api/hooks/razorpay/route.ts`.

**Refund flow:**
1. Webhook arrives with `event: "refund.processed"` + Razorpay `payment_id`
2. Route looks up the Medusa payment by matching `payment.data.id`
3. Resolves the linked order via `payment_collection`
4. If a formal `Return` exists → runs `send-order-refunded` workflow with the return ID
5. If no formal return (direct dashboard refund) → runs workflow with `isDirectRefund: true` and the paise amount

The `send-order-refunded` workflow was also updated to handle both paths:
- Normal path (`return.created` subscriber): `id` = Return ID
- Direct path (Razorpay webhook): `id` = Order ID + `isDirectRefund: true`

### 9. COD Twilio OTP Verification

Fully wired Twilio SMS into the COD payment provider.

**File:** `src/modules/cod-payment/service.ts`

**OTP flow:**
```
initiatePayment() — amount >= ₹3,000?
       ↓ YES
Generate 6-digit cryptographic OTP (crypto.randomInt)
Hash with per-session HMAC-SHA256 salt — never stored in plain text
Send SMS via Twilio REST API (no SDK dependency)
Store hash + salt + expiry + otp_required: true in payment session data
       ↓
Storefront shows OTP input → POST /store/cod/verify-otp
       ↓
authorizePayment() → blocks if otp_required=true && otp_verified !== true
```

**New API route:** `POST /store/cod/verify-otp`
- Request: `{ payment_session_id, otp }`
- Verifies HMAC hash, checks expiry, marks `otp_verified: true` in session data
- Errors: invalid OTP (400), expired OTP (400), session not found (404)

**Required env vars (backend):**
```
TWILIO_ACCOUNT_SID=
TWILIO_AUTH_TOKEN=
TWILIO_FROM_PHONE=   # E.164, e.g. +918888888888
```

**Graceful degradation:** if Twilio is unconfigured or the SMS fails, OTP is skipped
and flagged in session data (`otp_skipped_reason`) — checkout never blocked.

**Storefront — Callback page** (`src/app/auth/google/callback/page.tsx`):

1. Google redirects here with `?code=xxx&state=xxx`
2. `sdk.auth.callback("customer", "google", queryParams)` → JWT
3. Decode JWT: if `actor_id === ""` → new customer → create account → refresh token
4. Existing customer → token already valid
5. Redirect to `/{locale}/account`

`react-jwt` package installed in storefront for token decoding.

### 12. Email Verification After Signup

Implemented a complete post-signup email verification flow. Tokens are self-contained
(HMAC-SHA256 signed, nothing stored server-side) and expire after 24 hours.

**Location:** `src/api/store/auth/send-verification/route.ts` + `src/api/store/auth/verify-email/route.ts`

**Backend flow:**
```
Customer signs up
       ↓
Storefront fires POST /store/auth/send-verification (fire-and-forget, non-blocking)
       ↓
Backend generates token: base64url( customerId:email:expiresAt:hmac )
HMAC signed with JWT_SECRET — nothing stored server-side
       ↓
Sends "email-verification" template via Resend
       ↓
Customer clicks link → GET /store/auth/verify-email?token=xxx
       ↓
Backend: decode → check expiry → timing-safe HMAC compare → set customer.metadata.email_verified = true
```

**Storefront page:** `src/app/auth/verify-email/page.tsx`
- Client component at `/auth/verify-email` (outside `[countryCode]` — stable URL for email links)
- States: loading spinner → success (✅ auto-redirects to account after 2.5s) → expired/error with resend prompt
